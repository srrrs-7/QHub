# Testing Conventions

## Table-Driven Tests

All tests must use table-driven pattern:

```go
tests := []struct {
    testName string
    args     args
    expected expected
}{
    {
        testName: "descriptive name of test case",
        args:     args{ /* test inputs */ },
        expected: expected{ /* expected outputs */ },
    },
}

for _, tt := range tests {
    t.Run(tt.testName, func(t *testing.T) {
        // Test implementation
    })
}
```

## HTTP Handler Testing

Integration tests use real DB with transaction isolation:

```go
func TestGetHandler(t *testing.T) {
    q := testutil.SetupTestTx(t)  // transaction-isolated DB
    tt.setup(t, q)                // seed test data

    repo := task_repository.NewTaskRepository(q)
    handler := NewTaskHandler(repo).Get()

    req := httptest.NewRequest(http.MethodGet, "/tasks/"+id, nil)
    testutil.SetAuthHeader(req)   // set Bearer token
    w := httptest.NewRecorder()

    handler.ServeHTTP(w, req)

    if diff := cmp.Diff(http.StatusOK, w.Result().StatusCode); diff != "" {
        t.Errorf("status code mismatch (-want +got):\n%s", diff)
    }
}
```

For chi URL params in tests:
```go
rctx := chi.NewRouteContext()
rctx.URLParams.Add("id", taskID)
req = req.WithContext(context.WithValue(req.Context(), chi.RouteCtxKey, rctx))
```

## Database Testing

Use `testutil.SetupTestTx(t)` for transaction-wrapped tests (auto-rollback on cleanup):

```go
q := testutil.SetupTestTx(t)
repo := NewTaskRepository(q)

// Test repository methods with real database
got, err := repo.FindByID(context.Background(), id)
```

## Status Code Assertions — Be Specific

Assert the exact expected HTTP status code, not just "not 200". Weak assertions silently pass when errors produce incorrect status codes (e.g., 500 when 400 is expected):

```go
// WRONG — accepts any non-200 status, silently permits 500 on bad input
if resp.StatusCode == http.StatusOK {
    t.Errorf("expected error status, got %d", resp.StatusCode)
}

// CORRECT — asserts exactly what the spec requires
if diff := cmp.Diff(http.StatusBadRequest, resp.StatusCode); diff != "" {
    t.Errorf("status code mismatch (-want +got):\n%s", diff)
}
```

## Loop Variable Mutation Anti-Pattern

Do not mutate the table-driven test case struct (`tt`) inside the loop body. This is fragile if `t.Parallel()` is ever added, causing data races:

```go
// WRONG — mutates tt in place, races under t.Parallel()
for _, tt := range tests {
    t.Run(tt.testName, func(t *testing.T) {
        projID := createTestProject(t, q)
        tt.projectID = projID  // mutates shared loop variable
    })
}

// CORRECT — use a local variable
for _, tt := range tests {
    t.Run(tt.testName, func(t *testing.T) {
        projID := createTestProject(t, q)
        // use projID directly, don't assign back to tt
    })
}
```

## Environment Variable Tests and Parallelism

Tests that call `os.Setenv`/`os.Unsetenv` mutate global state. They must NOT use `t.Parallel()`:

```go
// If the test sets env vars, it cannot be parallel
func TestGetString(t *testing.T) {
    // Do NOT add t.Parallel() here — os.Setenv races
    t.Setenv("FOO", "bar")  // use t.Setenv for automatic cleanup
    // ...
}
```

Use `t.Setenv` (Go 1.17+) instead of `os.Setenv` + manual cleanup — it handles restore automatically and prevents accidental parallelism.

## Template Helper Function Testing

Pure Go helper functions defined in `.templ` files (e.g., `lintScoreClass`, `contentLines`, `availableTags`, `maxInt`) are compiled as regular Go functions and must be tested like any other function. Create a `templates/helpers_test.go` (or equivalent) covering all 6 test categories:

| Function | Key Cases to Cover |
|---|---|
| `lintScoreClass(score int)` | Boundary: 79 vs 80, 59 vs 60; negative scores |
| `contentLines(content string)` | Empty string, Windows `\r\n` endings, nil-equivalent |
| `availableTags(all, current)` | Nil slices, empty slices, all tags already assigned |
| `maxInt(a, b int)` | Equal values, negative, min/max int |

## Partial Handler Tests for New Components

When new templ components are added (`LintResultCard`, `TextDiffCard`, `SemanticDiffCard`, `VersionCompareCard`, etc.), verify they are exercised by at least one existing or new partial handler test. Adding a component without updating handler tests leaves rendering paths untested.

## Test Organization

- Unit tests: Test value objects and domain logic
- Integration tests: Test repositories and handlers with real DB (requires `docker compose up -d db`)
- Table-driven tests: Group related test cases together
- Test naming: Use descriptive names that explain the scenario

## Build Artifact Hygiene

Never commit compiled test binaries or coverage profiles:
- Add `*.test` and `coverage.out` to `.gitignore`
- These are generated by `go test -c` and `-coverprofile` respectively
- They bloat the repository and are not version-controlled artifacts

## Running Tests

```bash
# All tests (requires DB)
make test

# Single package
cd apps/api && go test ./src/routes/tasks/

# Single test
cd apps/api && go test -run TestListHandler ./src/routes/tasks/

# With coverage
cd apps/api && go test -cover ./...

# Find functions below 80%
cd apps/api && go test -coverprofile=c.out ./... && \
  go tool cover -func=c.out | awk -F'[ \t]+' '$NF+0 < 80.0 {print $0}'
```
