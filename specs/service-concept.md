# Service Concept

## One-Liner

**PromptLab — プロンプトの価値を、結果で証明し、ナレッジで進化させる。**

---

## Core Insight (コアとなる洞察)

### 洞察 1: プロンプトの価値は出力にある

GitHub はコードという「成果物」を管理する。コードの価値はコード自体に内在する。
しかし、プロンプトは違う。

**プロンプトの価値は、プロンプト自体にはない。それが生み出す出力の品質にある。**

```
コード:    書いたものが動く → 品質 = コード自体の正しさ
プロンプト: 書いたものが LLM に渡される → 品質 = 出力の品質
```

### 洞察 2: ナレッジは使われて初めて価値になる

プロンプト改善の過程で蓄積されるデータ（バージョン履歴、品質スコア、成功/失敗パターン）は、
放置すればただの記録。しかし、**業界知識と組み合わせてチャットで提供すれば、コンサルタントになる。**

```
蓄積データ (静的)  →  業界特化コンサルチャット (動的)
                        │
                        ├─ 「御社のプロンプト v5 は正確性が v3 比で 11% 改善。
                        │    次は安全性制約の追加を推奨。医療業界の top パターンでは...」
                        │
                        └─ 「同業他社の匿名ベンチマークでは平均スコア 85。
                             御社は 78。差分の原因はここにあります...」
```

## プロダクトの二層構造

```
┌─────────────────────────────────────────────────────────┐
│  Layer 2: Industry Consulting Chat (業界特化コンサル)     │
│  ┌───────────────────────────────────────────────────┐  │
│  │  蓄積データ × 業界知識 × LLM = AI コンサルタント   │  │
│  │                                                   │  │
│  │  ・組織のプロンプト改善提案                         │  │
│  │  ・業界ベストプラクティス                           │  │
│  │  ・匿名ベンチマーク比較                            │  │
│  │  ・品質改善のアクションプラン                       │  │
│  └───────────────────────────────────────────────────┘  │
│         ▲ データが溜まるほど賢くなる                     │
│         │                                               │
│  Layer 1: Prompt Management Platform (基盤)              │
│  ┌───────────────────────────────────────────────────┐  │
│  │  Write → Deploy → Observe → Learn → Improve       │  │
│  │                                                   │  │
│  │  ・バージョン管理 + Semantic Diff                   │  │
│  │  ・ログ ingestion + 品質評価                        │  │
│  │  ・Prompt Linting + 分析ダッシュボード              │  │
│  └───────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

**Layer 1** が基盤としてデータを蓄積し、**Layer 2** がそのデータを活用して付加価値を生む。
Layer 1 だけでも十分に価値があるが、Layer 2 が加わることで**データのネットワーク効果**が生まれる。

## Outcome-Driven Prompt Management (Layer 1)

プロンプトを「書く」ことではなく、「結果を出す」ことにフォーカスする。

```
    ┌──────────────────────────────────────────────┐
    │                                              │
    │    Write ──► Deploy ──► Observe ──► Learn    │
    │      ▲                                │      │
    │      └────────── Improve ◄────────────┘      │
    │                                              │
    └──────────────────────────────────────────────┘
```

### Write (書く)
- テンプレート変数によるパラメータ化
- Prompt Linting で品質チェック
- Semantic Diff で「何を変えたか」を意味レベルで理解

### Deploy (届ける)
- Draft → Review → Production のシンプルなライフサイクル
- SDK 経由で最新バージョンを取得
- ロールバック可能

### Observe (観測する)
- SDK / API 経由で実行ログを自動送信
- リクエスト / レスポンス / メタデータを構造化保存

### Learn (学ぶ)
- 手動 / 自動の品質評価
- バージョン間の統計的比較
- コスト・レイテンシ・品質のトレードオフ可視化

### Improve (改善する)
- 低スコアパターンの自動発見
- Natural Language Changelog
- 評価スイートによるリグレッション防止

## Industry Consulting Chat (Layer 2)

### コンセプト

蓄積されたプロンプトデータ・品質データと、業界特化ナレッジを組み合わせた
**RAG ベースの AI コンサルタント**。

「データに基づいてプロンプトの改善をアドバイスしてくれる、業界に詳しい専門家」をチャットで提供する。

### 知識ソース

```
┌─────────────────────────────────────────────────────────┐
│                 Consulting Chat Engine                    │
│                                                         │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │ Org Data     │  │ Industry KB  │  │ Platform     │  │
│  │              │  │              │  │ Insights     │  │
│  │ ・プロンプト  │  │ ・業界規制   │  │ ・匿名統計   │  │
│  │   履歴       │  │ ・ベスト     │  │ ・成功パターン│  │
│  │ ・品質推移   │  │   プラクティス│  │ ・失敗パターン│  │
│  │ ・ログデータ │  │ ・用語集     │  │ ・ベンチマーク│  │
│  │ ・評価結果   │  │ ・コンプラ   │  │              │  │
│  │              │  │   要件       │  │              │  │

```

| 知識ソース | 内容 | アクセス範囲 |
|-----------|------|-------------|
| **Org Data** | 組織のプロンプト、バージョン、ログ、評価 | 自組織のみ |
| **Industry KB** | 業界固有の規制、ベストプラクティス、用語 | 契約業界 |
| **Platform Insights** | 匿名化されたクロス組織の統計・パターン | オプトイン組織 |

### 対応業界 (初期)

| 業界 | ナレッジの例 |
|------|-------------|
| **ヘルスケア** | HIPAA 準拠プロンプト、医療用語の正確性、患者情報の取り扱い |
| **法律** | 法的推論パターン、判例引用形式、免責事項の必須要素 |
| **金融** | 規制コンプライアンス、リスク評価、金融アドバイスの制約 |
| **カスタマーサポート** | トーン管理、エスカレーションパターン、多言語対応 |
| **教育** | 年齢適切性、学習段階に応じた説明レベル、アクセシビリティ |
| **EC/小売** | 商品説明の最適化、レコメンデーション、レビュー分析 |

### チャット機能の具体例

#### 1. プロンプト改善提案

```
User: customer-support-bot の品質が v4 から下がっています。原因は？

Chat: v4 での変更点を分析しました。

📊 品質変化:
  - 正確性: 85.2 → 79.1 (-6.1)
  - 有用性: 88.0 → 82.3 (-5.7)
  - 安全性: 95.0 → 96.2 (+1.2)

🔍 原因分析:
  v4 で安全性制約を強化した結果、回答が保守的になりすぎています。
  特に「返品ポリシー」関連の質問で、具体的な手順を省略する傾向が見られます。

💡 改善提案:
  安全性制約を維持しつつ、以下の修正を推奨します:
  - "Be specific about actionable steps" を追加
  - 返品手順のテンプレートを user_template に含める

  [提案を v5 として作成する]
```

#### 2. 業界ベンチマーク比較

```
User: 当社のヘルスケアチャットボットは業界水準と比べてどうですか？

Chat: ヘルスケア業界のベンチマーク比較です。

📊 御社 vs 業界中央値 (匿名集計、オプトイン組織 47 社):

  正確性:   御社 82.3 | 業界 85.1 | 差 -2.8
  安全性:   御社 96.5 | 業界 94.2 | 差 +2.3 ✅
  有用性:   御社 78.0 | 業界 83.7 | 差 -5.7
  コスト:   御社 $2.85/1K | 業界 $3.20/1K | 差 -$0.35 ✅

💡 改善の優先順位:
  1. 有用性 (+5.7 必要): 回答に具体的なアクションステップを含めることで改善。
     業界トップの組織は「次に何をすべきか」を必ず回答に含めています。
  2. 正確性 (+2.8 必要): 医療用語の定義を system prompt に含めることで
     曖昧な回答を減らせます。

  [改善プランを生成する]
```

#### 3. コンプライアンスチェック

```
User: 新しいプロンプトが HIPAA に準拠しているか確認してください。

Chat: v6 の HIPAA 準拠チェック結果:

  ✅ PHI (個人健康情報) の直接参照なし
  ✅ データ保持に関する制約あり
  ⚠️ 患者の同意確認フローが不十分
    → "Always verify patient consent before discussing treatment details" を追加推奨
  ❌ 緊急時の免責事項が欠落
    → "This is not a substitute for professional medical advice" の追加が必須

  準拠率: 75% (最低要件: 90%)
  [修正提案を適用して v7 を作成する]
```

## コンセプトの四本柱

### 1. Version as Truth (バージョンが真実)

> 「本番で動いているプロンプトがどれか分からない」を根絶する。

- イミュータブルなバージョン番号
- 実行ログは必ずバージョンに紐付く
- Draft → Review → Production のシンプルなライフサイクル

### 2. Quality as Metric (品質をメトリクスに)

> 「このプロンプト、なんとなく良い気がする」を排除する。

- 品質を 0-100 スコアで数値化
- バージョン単位の品質推移を時系列表示
- 統計的有意差検定で改善を証明

### 3. Loop as Speed (ループを速く)

> 改善サイクルを 1 週間から 1 時間に短縮する。

- Write → Deploy → Observe → Learn → Improve を 1 画面で完結
- Playground でその場でテスト、即バージョン化
- 評価スイートで自動リグレッションチェック

### 4. Knowledge as Service (ナレッジをサービスに)

> 蓄積データを眠らせない。業界知識と合わせて、コンサルタントにする。

- 組織固有データ × 業界ナレッジ × プラットフォーム統計
- チャットベースで即座にアドバイス
- データが溜まるほど賢くなるフライホイール

## ペルソナ別の価値提案

### 開発者

**Before**: プロンプトを定数に書き、変更のたびに手戻り。業界の規制も手探り。
**After**: バージョン管理 + 品質データ + 業界コンサルで、データドリブンに改善。

**キラーフィーチャー**: SDK でログ自動送信 → データ蓄積 → コンサルチャットが改善提案

### プロダクトマネージャー

**Before**: 品質の状況が不透明。業界ベンチマークもない。
**After**: ダッシュボードで品質トレンド + 業界比較が一目瞭然。

**キラーフィーチャー**: 業界ベンチマーク比較 + コンサルチャットの改善プラン

### ドメインエキスパート (医療・法律・金融)

**Before**: LLM の技術的な使い方は分からないが、業界知識は豊富。
**After**: チャットに業界要件を伝えるだけで、準拠したプロンプトの改善提案を受けられる。

**キラーフィーチャー**: コンプライアンスチェック + 業界テンプレート

## プロダクト原則

1. **Observe First, Optimize Second**
   まず観測の基盤を作る。最適化はデータが溜まってから。

2. **Simple by Default, Powerful when Needed**
   基本操作は直感的に。高度な分析は必要な人だけ使う。

3. **SDK が入口、UI が分析の場、Chat がアクションの場**
   SDK でデータが溜まり、UI で理解し、Chat で次のアクションを得る。

4. **Immutable History**
   バージョンもログも評価も不変。改ざんできない履歴が信頼の基盤。

5. **Natural Language Native**
   自然言語を第一級市民として扱う。コード管理の概念を無理に当てはめない。

6. **Data Compounds**
   データは複利で価値が増す。蓄積するほどコンサルが賢くなる設計。

## MVP の核心

検証すべき仮説:

> **H1: 「プロンプトの実行ログと品質スコアをバージョン単位で可視化すれば、
> 開発者はプロンプト改善サイクルを自律的に回せるようになる」**

> **H2: 「蓄積データと業界知識を組み合わせたチャットは、
> プロンプト改善の速度と品質を既存ワークフローより有意に向上させる」**

MVP 機能:
1. プロンプト + バージョン管理 (Write)
2. SDK でログ送信 (Deploy + Observe)
3. スコア付き評価 (Learn)
4. バージョン別ダッシュボード (Improve の動機付け)
5. **基本コンサルチャット** — 組織データに基づく改善提案 (Knowledge as Service)

H1 は Layer 1 で検証。H2 は Layer 2 で検証。両方同時に検証する。
